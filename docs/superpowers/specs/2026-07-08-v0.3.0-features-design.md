# v0.3.0 Features — Design / PRD

Source: GitHub issue #5 ("v0.3.0 Features"). Status: approved for implementation.
Date: 2026-07-08.

## Overview

v0.3.0 adds three independent, user-facing features to simpleGals, each gated by
a project-level boolean:

1. **Social media previews** (extends the partial OG/Twitter work shipped in v0.2.0)
2. **Camera metadata** on item pages (new)
3. **Gallery download zip** of full-size originals (new)

All three are configurable in `simpleGal.json` and in the `sgui` gallery settings
panel (Ctrl+G).

## Prior art already in the tree

- `item.html.j2` and `page.html.j2` already emit `og:*` and `twitter:*` tags, but
  `og:image` points at the 600x450 thumbnail, there is no on/off toggle, and the
  title/description rules do not yet match the issue. Feature 1 is therefore an
  extension, not a greenfield build.
- `processor.py` already runs Pillow with EXIF orientation handling and uses
  `piexif` for copyright injection. Feature 2 builds on the existing Pillow path.
- `workers/progress.py` + `tui/progress_bar.py` provide a progress mechanism that
  Feature 3 hooks into.

## Cross-cutting changes

### Config (`core/config.py`, `ProjectConfig`)

Three new booleans, snake_case for consistency with existing keys (`site_url`,
`file_panel_width`, etc.). JSON keys map 1:1 to dataclass fields as they do today.

| Field | Type | Default | Gates |
|---|---|---|---|
| `social_previews` | `bool` | `True` | OG/Twitter tags and `_og.jpg` generation |
| `exif_display` | `bool` | `True` | EXIF metadata block on item pages |
| `gallery_zip` | `bool` | `False` | Album zip and download button |

- `load_project_config` / `save_project_config` round-trip them automatically
  (they iterate `__dataclass_fields__`).
- `sgui` settings panel (Ctrl+G) gains three toggles.
- `settings_hash` is unchanged: none of these settings alter the pixels of the
  full/thumb/display outputs, so they must not invalidate those caches. The
  `_og.jpg` artifact is tied to output staleness (see Feature 1), and template
  re-render happens on every build regardless.

### Sidecar (`core/metadata.py`, `ImageSidecar`)

Two new optional fields, both cached and refreshed only when the image's output
is stale:

- `og: OgMeta | None` where `OgMeta{ path: str, generated_at: str }`
- `exif: dict | None` (normalized EXIF; see Feature 2)

`load_sidecar` / `save_sidecar` gain (de)serialization for both, mirroring the
existing `thumb` / `output` handling.

### Accessibility and focus (cross-cutting)

- `style.css` has no `:focus` / `:focus-visible` rule today. Add a site-wide
  `:focus-visible { outline:2px solid #ff69b4; outline-offset:2px; }` (reusing the
  existing pink accent) so the new download button and existing links show a clear
  keyboard-focus ring.
- Add an `aria-label` to the existing unlabeled 💾 item-download link
  (`aria-label="Download full-size image"`) while download semantics are in scope.
  Any new icon-only control must carry its own `aria-label`; do not rely on the
  hover-only `.share-hint` / `.back-hint` label pattern for accessibility. (The zip
  download button has a permanent visible text label, so it needs nothing beyond
  the focus ring.)

## Feature 1 — Social media previews

### Artifact

New per-image artifact `out/<stem>_og<ext>`:

- The preview mirrors the source container, exactly like the `_thumb`/`_display`
  outputs: a JPEG source yields `<stem>_og.jpg`, a PNG source yields
  `<stem>_og.png` with its transparency preserved. It is saved through the same
  `_format_save_kwargs` path, so only container-applicable options are applied
  (ICC for both; no `convert("RGB")` flatten). This avoids the black-box artifact
  a forced JPEG flatten produces on transparent PNG sources.
- Longest edge scaled to 1200px (aspect preserved), quality ~80. (Raised from the
  issue's 300px: 300px falls below Twitter/X `summary_large_image` and Facebook
  card minimums for portrait/panorama images and triggers a silent downgrade to a
  small, soft card. At 1200px/q80 the file is ~80-250KB, negligible next to the
  2048px `_display.jpg`.) No aspect-ratio floor or crop is applied (we never crop
  content); an extreme panorama could still have a short edge below some platforms'
  comfort threshold, which is an accepted tradeoff.
- Generated in `processor.generate_output()` alongside thumb/display, only when
  `social_previews` is true.
- Recorded in the sidecar as `OgMeta`; regenerated whenever the output is stale.
- Pruned alongside the other artifacts in `prune_removed_sources`.

### Templates

- `item.html.j2`: `og:image` and `twitter:image` point at `<stem>_og<ext>`
  (absolute, prefixed with `site_url`) instead of the 600x450 thumbnail. The
  template renders `image.og_path`, which already carries the correct extension.
- `page.html.j2`: index/paginated pages use the first image's `_og.jpg`.
- All `og:*` / `twitter:*` tags are wrapped in `{% if social_previews %}`.
- Image URL tags remain gated on `site_url` (OGP requires an absolute URL).
- **Title:** `og:title` / `twitter:title` become `{gallery title}{sep}{item name}`,
  reusing the separator already used in the existing `<title>` so the page is
  internally consistent.
- **Description precedence:**
  - item pages: caption, else `description`, else gallery title
  - index pages: `description`, else gallery title

### Behavior when `site_url` is unset

Base tags (`og:type`, `og:site_name`, titles, descriptions) still render when
`social_previews` is true; the image/url tags are omitted, matching current
behavior. `_og.jpg` is still generated (cheap, and lets the author add `site_url`
later without a rebuild).

## Feature 2 — Camera metadata

### Extraction (`core/exif.py`, new module)

`extract_exif(source: Path) -> dict | None`:

- Reads EXIF via Pillow `Image.getexif()` + `ExifTags` (no new dependency;
  `piexif` is also already available if richer parsing is needed).
- Returns a normalized dict with any of these keys that are present:

  | Key | Source tag(s) | Rendered example |
  |---|---|---|
  | `camera` | Make + Model | `NIKON Z 7` |
  | `lens` | LensModel (+ LensMake) | `24-70mm f/2.8` |
  | `exposure` | FNumber + ISO + ExposureTime | `f/5.6 · ISO 100 · 1/125s` |
  | `focal_length` | FocalLength | `50mm` |
  | `flash` | Flash | `Fired` / `Did not fire` |
  | `exposure_comp` | ExposureBiasValue | `-0.7 EV` (omitted when 0) |
  | `white_balance` | WhiteBalance | `Auto` / `Manual` |
  | `metering` | MeteringMode | `Pattern` |

- Returns `None` when no usable tags exist (e.g. PNG or stripped JPEG).

### Caching and data flow

- Extraction runs during `build()` when the output is (re)generated or the sidecar
  has no `exif` yet; the result is stored in the sidecar.
- The build's record-assembly loop reads `sidecar.exif` and passes it into the
  render record for each image.

### Template (`item.html.j2`)

- Inserted **between** `nav.item-nav` and `<footer>`.
- Rendered **only** when `exif_display` is true **and** the image has EXIF data
  (the "hide the block" decision). Images with no EXIF render normally with no
  block.
- **Not a `<table>`, and not a fixed 2x4.** Real files often fill only 3-5 of the
  8 fields (prime lenses have no zoom `focal_length`; phone JPEGs often lack
  white balance / metering / exposure comp), so a fixed 8-slot grid would show
  visible blank cells. Render a definition list of **only the present fields**:
  ```html
  <dl class="exif">
    <div class="exif-field"><dt>Camera</dt><dd>NIKON Z 7</dd></div>
    <!-- ...only fields that exist... -->
  </dl>
  ```
- Lay it out with an auto-flowing grid so it reflows with no media queries (there
  are none in `style.css`):
  `display:grid; grid-template-columns:repeat(auto-fit, minmax(150px, 1fr)); gap:0.5rem 1.5rem;`
  3 fields lay out as 3, 8 as 8, collapsing to 2/1 columns as the viewport narrows.
- Style to the borderless, muted `.item-meta` type scale, not table chrome:
  labels `font-size:0.7rem; text-transform:uppercase; letter-spacing:.04em;
  color:#666;`, values `font-size:0.85rem; color:#ccc;`. The only separator is a
  single `border-top:1px solid #333;` on the container, plus `margin-top:1.5rem`
  so it reads as distinct from the prev/next buttons directly above it.

## Feature 3 — Gallery download zip

### Archive builder (`core/archive.py`, new module)

`build_zip(out_dir, included_names, zip_path, progress_cb=None) -> tuple[int, int]`
returning `(file_count, total_bytes)`:

- Uses `zipfile` with `ZIP_STORED` (0 compression; images do not compress).
- Contains the **processed `out/` originals** (the `out/<name>.jpg` files that
  carry injected EXIF copyright), images only.
- Adds files one at a time, invoking `progress_cb` per file so the UI can show
  item-by-item progress.

### Incremental rebuild

- A manifest of `(name, sha256)` for the included originals is written to
  `.meta/gallery-zip.json`.
- The zip is (re)built only when that manifest changes or the zip file is missing.
  Otherwise the existing archive and its recorded size are reused.

### Naming

- `slugify(title) + ".zip"` in `out/` (e.g. `My Gallery` -> `My_Gallery.zip`),
  with illegal filename characters sanitized. (A future release will expose this
  as a template variable, per the issue; out of scope here.)

### Build integration (`core/gallery.py`)

- When `gallery_zip` is true, after outputs are generated: compute the manifest,
  rebuild if stale, and obtain `(count, size)`.
- A new zip phase is added to `ProgressState` (`zip_total`, `zip_done`) and driven
  through the existing `progress_callback`, so both the `simpleGals` CLI and `sgui`
  show progress. If per-item progress is ever unavailable, a text notice is logged
  instead.
- Log line reports count and final size, e.g.
  `Zipped 42 images -> My_Gallery.zip (318 MiB)`.

### Download button (`page.html.j2`)

- A rounded-rectangle **outline chip** matching the existing `nav.pagination`
  ghost-buttons (`border:1px solid #444; border-radius:4px`), not a solid-filled
  CTA, differentiated only by a tinted border/background (e.g. the existing
  `#ff69b4` accent or a subtle tint). A filled button would clash with the site's
  ghost-button language.
- Label `Download all (<size>)` where size uses
  `bitmath...best_prefix().format("{value:.0f} {unit}")`, integer only, e.g.
  `Download all (3 MiB)`. "all" disambiguates the whole-gallery zip from the
  per-image 💾 download already on item pages.
- The template receives `gallery_zip` (filename) and `gallery_zip_size` (preformatted
  string).
- **Placement (both page types) via a shared Jinja snippet** (macro/include) so
  the label/size formatting has a single source of truth:
  - Index / paginated pages: next to the "View all" link in `nav.pagination`, in
    DOM order (so tab order matches visual order; no CSS `order` reshuffling).
  - `all.html`: it has no pagination nav, so give it its **own thin chip-row under
    the header** (a `nav.pagination`-style wrapper containing just the button, no
    page-number chips), rather than dropping a small chip into the 1.6rem `<h1>`
    header row where it would look mis-sized and could overflow on narrow screens.

## Versioning refactor (single source of truth)

Adopt the dynamic-version setup used by the sibling `bitmath` project so the
version lives in exactly one place, ending the current drift (`pyproject.toml`
declares `0.2.0` while `simplegals/__init__.py` and the installed metadata still
read `0.1.0`).

### `VERSION` file (new, repo root)

- Single source of truth. Contains only the bare version string, e.g. `0.2.0`.
- **No trailing newline** and no other characters. (bitmath's `VERSION` ends in
  `\n`; ours must not.)
- Initialize to `0.2.0` (current released version). Must be committed / git-tracked
  so hatchling can resolve the version in a source build.

### `pyproject.toml`

- In `[project]`, remove the hardcoded `version = "0.2.0"` and add
  `dynamic = ["version"]`.
- Add a hatchling version source pointing at the file:

  ```toml
  [tool.hatch.version]
  path = "VERSION"
  pattern = '(?P<version>.+)'
  ```

  (`.+` captures the version and does not match a newline, so the file works with
  or without a trailing newline; we keep it newline-free per above.)

### `simplegals/__init__.py`

Derive `__version__` from installed package metadata rather than hardcoding it, so
it stays single-sourced from the build:

```python
from importlib.metadata import PackageNotFoundError, version as _version

try:
    __version__ = _version("simplegals")
except PackageNotFoundError:  # running from a source tree without an install
    __version__ = "0.0.0+unknown"
```

(bitmath omits a module `__version__` entirely; we keep one but derive it. Nothing
in the codebase currently reads `__version__`, so this is a safe change.)

### Verification

- After the change, run `pip install -e .` to refresh the stale editable metadata,
  then confirm `python -c "import simplegals; print(simplegals.__version__)"` and
  `hatch version` (or a build) report `0.2.0`.

### Tag / version prefix convention

One bare PEP 440 version string is used in **every** context, with **no `v`
prefix** anywhere:

| Context | Value |
|---|---|
| `VERSION` file | `0.3.0` |
| `pyproject.toml` / PyPI | `0.3.0` (PEP 440 forbids a leading `v` in the canonical version) |
| Git tag | `0.3.0` |
| GitHub Release name | `0.3.0` |

Rationale: the package version is immutably bare, so bare tags are the only way to
have one identical string everywhere. This also matches the existing tag history
(`0.1.1`, `0.1.2`, both bare). A `v` prefix would only reintroduce a permanent
tag-vs-package mismatch plus the normalization logic needed to reconcile it. This
deliberately diverges from bitmath (which tags with `v`): we copy bitmath's
`VERSION`-file mechanism, not its tag convention.

### Publish-time guard (`.github/workflows/publish.yml`)

Add a step to the existing release-triggered publish job that fails the release
when the tag does not exactly equal the `VERSION` file. Because both are bare, it
is a plain string comparison with no `v`-stripping:

```yaml
- name: Verify tag matches VERSION
  run: |
    TAG="${GITHUB_REF#refs/tags/}"
    [ "$TAG" = "$(cat VERSION)" ] || {
      echo "::error::Release tag '$TAG' does not match VERSION '$(cat VERSION)'. Bump VERSION before tagging."
      exit 1
    }
```

Cheap insurance against publishing a wheel whose version does not match the release
tag (the bump-order footgun). The publish workflow keeps triggering on
`release: published`; nothing keys off a tag pattern.

### Release bump procedure

- Bumping becomes a one-line edit to `VERSION` (`0.2.0` -> `0.3.0`); no other file
  changes. Reinstall/rebuild, then tag `0.3.0` and publish the GitHub Release named
  `0.3.0` (bare, no `v`).
- **Update `CLAUDE.md`** with a short "Versioning" note documenting that `VERSION`
  is the sole source, the bump is a one-line edit, and the one-bare-string / no-`v`
  rule (git tags and GitHub Releases included), so future maintenance tasks
  auto-detect the procedure. (This replaces the earlier idea of syncing two files.)

### 0.3.0 release step

- Once the features in this PRD are implemented and verified, set `VERSION` to
  `0.3.0` as the final release action.

## Testing

- **config:** round-trip of the three new booleans; defaults correct.
- **exif:** extraction from a fixture with known EXIF; `None` from a fixture with
  none; individual-field normalization (exposure string, flash, exposure comp
  omitted when 0).
- **social:** `_og.jpg` longest edge == 1200px; tags gated by `social_previews`;
  title/description precedence.
- **archive:** `ZIP_STORED` used; contents are the `out/` originals only;
  incremental skip when manifest unchanged; integer size formatting via bitmath
  `best_prefix`.
- **templates:** EXIF block gated by `exif_display` + presence and renders only
  present fields (no blank cells, no `<table>`); download button labeled
  `Download all (...)` and gated by zip presence; og image points at `_og.jpg`.
- **accessibility:** `:focus-visible` outline present; the 💾 item-download link
  and any icon-only control carry an `aria-label`.
- **sgui:** the three settings toggles read/write config.

## Out of scope (deferred)

- Templatable zip archive name (issue notes this as future work).
- Per-image opt-out of social previews / EXIF (release-level toggles only).
